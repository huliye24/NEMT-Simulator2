# 异步协作流程 - 邮件式收件箱

> 基于 Notion "四角色异步交互流程" 设计

---

## 核心概念

### 邮件式异步工作流

每个角色拥有专属的"数据库视图"，只显示自己负责的任务，就像一个**收件箱**。

```
┌─────────────────────────────────────────────────────────────┐
│                    Notion Task Inbox                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │ CEO     │  │ CTO     │  │ PM      │  │ Tech    │   │
│  │ 收件箱  │  │ 收件箱  │  │ 收件箱  │  │ Lead    │   │
│  │         │  │         │  │         │  │ 收件箱  │   │
│  │ Task#1 │  │ Task#5 │  │ Task#8 │  │ Task#12│   │
│  │ Task#2 │  │ Task#6 │  │ Task#9 │  │ Task#13│   │
│  │ ...    │  │ ...    │  │ ...    │  │ ...    │   │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 收件箱设置步骤

1. 进入 Task Inbox 数据库
2. 点击 `+ Add a View` → 选择 `Board` 或 `Table`
3. 命名视图为 `📥 [角色名] 收件箱`
4. 点击 `Filter` → 添加过滤条件：`负责人` `Contains` `[角色名]`
5. 保存

## 四张核心表设计

### 表 1: Task Inbox (任务收件箱)

| 字段 | 类型 | 说明 |
|------|------|------|
| 任务名称 | Title | 任务标题 |
| 负责人 | Person | 当前负责人 |
| 状态 | Select | 待认领/进行中/需审核/已完成 |
| 优先级 | Select | P0/P1/P2/P3 |
| 任务类型 | Select | 愿景检查/开发任务/决策请求/代码审查 |
| 描述 | Text | 详细描述 |
| 关联决策 | Relation | 关联 Decision Log |

**自动化规则**:
- 当 `负责人` 变更时 → 通知新负责人
- 当 `状态` 变为 `需审核` 且 `任务类型` 为 `决策请求` → 自动指派给 CEO

---

### 表 2: Decision Log (决策记录)

| 字段 | 类型 | 说明 |
|------|------|------|
| 决策标题 | Title | 决策摘要 |
| 日期 | Date | 决策日期 |
| 决策人 | Person | 做出决策的人 |
| 愿景对齐 | Checkbox | 是否已做愿景对齐检查 |
| 分析摘要 | Text | 决策分析 |
| 结果 | Select | 通过/拒绝/条件通过 |
| 关联任务 | Relation | 关联 Task Inbox |

**自动化规则**:
- 新建 `决策记录` 时 → 自动创建任务给 PM："更新相关任务"

---

### 表 3: Vision Milestones (愿景里程碑)

| 字段 | 类型 | 说明 |
|------|------|------|
| 里程碑名称 | Title | 里程碑标题 |

| 状态 | Select | 规划中/开发中/已完成 |
| 验收状态 | Select | 待验收/验收通过/验收失败 |

| 字段 | 类型 | 说明 |
|------|------|------|

| 评分 | Number | 愿景对齐评分 (1-5) |

## 标准工作流实例

### 场景：PM 发现新需求——给回测结果加折线图

#### Step 1: 任务触发 (PM 视角)

PM 在 Task Inbox 创建任务：
```
任务名称: 评估回测结果可视化需求
负责人: PM
任务类型: 愿景检查
描述: 有用户反馈想看权益曲线图，虽然 V1 不做 UI，但图对调试有帮助。
```

PM 对照 VISION.md，看到边界写着："不做 Web 前端（V1）"

PM 在任务描述中勾选愿景对齐检查：
```
愿景对齐检查:
- [ ] 是否在边界之外？ → 是（Web前端相关）
- [ ] 有没有更简单的方案？ → 有，导出 CSV 用 Excel 看
```

**检查结果**: 未通过

#### Step 2: 转交决策 (PM → CEO)

PM 将任务更新：
```
状态: 需审核
负责人: CEO
任务类型: 决策请求
```

Notion 自动通知 CEO："有一个潜在偏离愿景的请求需要裁决"

#### Step 3: 异步裁决 (CEO 视角)

CEO 打开"CEO 收件箱"视图，看到这条任务

CEO 在评论中 @CTO："你觉得为了调试加一个本地生成的 PNG 算前端吗？"

CTO 分析后给出建议

#### Step 4: CTO 创建决策记录

CTO 在 Decision Log 新建记录：
```
决策标题: V1 阶段回测可视化方案
愿景对齐分析:
  - 原则4（架构优先于功能）: CSV导出更简单
  - 边界（不做Web）: 严格禁止
结果: 拒绝 V1 内置图表；V2 再评估
```

#### Step 5: 执行与验收

PM 收到通知，将任务转化为具体开发任务：
```
任务名称: 实现回测结果 CSV 导出功能
负责人: Tech Lead
优先级: P1
关联决策: V1 阶段回测可视化方案
```

Tech Lead 开发完成，标记"需审核"

CTO 代码审查通过，标记"已完成"

## 每日/每周仪式

### 每日站会 (5分钟，异步)

**时间**: 每天 9:00 AM

**动作**: 每个人在 Weekly Vision Sync 页面评论区发一条：
```
[Agent名] - YYYY-MM-DD
今天主要处理: Task #123, #124
阻塞: 无 / 等待 CEO 审批 #125
```

### 每周愿景对齐会 (15分钟，CEO主持)

**时间**: 周一 10:00 AM

**议程**:
1. CEO 检查上周任务是否有未做愿景检查的
2. 查看 Decision Log 最近决策是否违背 VISION
3. 给上周打分，记录纠正措施

### 里程碑验收 (里程碑完成后)

**触发**: PM 将 Vision Milestones 状态改为"开发中完成"

**动作**: CEO + CTO 打开里程碑详情，执行验收清单：
```
- [ ] 代码是否引入了新依赖？
- [ ] 是否生成了新的未规划子项目？
- [ ] 核心原则是否受损？
```

**结论**: 只有全部通过，才改为"验收通过"

## 模拟邮件已读/未读

**问题**: 如何模拟邮件的已读/未读状态？

**方案1**: 使用"状态"字段
- 约定：`状态 = 待认领` 即为"未读邮件"

**方案2**: 添加"最后查看时间"字段
- 每次打开视图时自动更新

**推荐**: 方案1，最简单有效
