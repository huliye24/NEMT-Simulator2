## 背景与问题

目标：打造能24小时不间断运行的加密货币策略模拟系统，需要一套精密的技术栈来支撑**数据、回测、策略和部署**四个核心环节。

**核心问题**：如何选择技术方案，平衡开发效率与系统性能？

---

## 候选方案对比

### 方案 A：Freqtrade 路线

| 维度 | 评分 | 说明 |
|------|------|------|
| 上手难度 | ⭐⭐ | 有成熟文档和社区 |
| 回测性能 | ⭐⭐⭐ | Python 实现，中等性能 |
| 实盘一致性 | ⭐⭐ | 需要适配层 |
| 定制化 | ⭐⭐⭐ | 开源，可深度定制 |

**优点**：快速上线，内置回测和实盘
**缺点**：与 NEMT 架构不兼容，需重写信号生成逻辑

---

### 方案 B：NautilusTrader 路线

| 维度 | 评分 | 说明 |
|------|------|------|
| 上手难度 | ⭐⭐⭐⭐⭐ | Rust 核心，学习曲线陡 |
| 回测性能 | ⭐⭐⭐⭐⭐ | Rust + 高性能 |
| 实盘一致性 | ⭐⭐⭐⭐⭐ | 回测=实盘代码 |
| 定制化 | ⭐⭐ | 框架约束较多 |

**优点**：工业级性能，回测实盘一致
**缺点**：学习成本高，NEMT 核心算法难以集成

---

### 方案 C：自研 + CCXT（推荐）✅

| 维度 | 评分 | 说明 |
|------|------|------|
| 上手难度 | ⭐⭐⭐ | 需要一定开发量 |
| 回测性能 | ⭐⭐⭐ | 可优化 |
| 实盘一致性 | ⭐⭐⭐⭐ | 同一套代码 |
| 定制化 | ⭐⭐⭐⭐⭐ | 完全掌控 |

**优点**：
- NEMT 已有独特的 NLS 信号生成逻辑，可直接复用
- CCXT 提供统一的数据和交易接口（100+ 交易所）
- 保持架构一致性
- 完全掌控，可深度定制

**缺点**：
- 需要自研回测引擎
- 开发周期较长

---

## 推荐方案及理由

**CTO 推荐：方案 C - 自研 + CCXT**

理由：
1. NEMT 核心算法（NLS 方程 + 四相位状态机）是独特资产，无法直接复用 Freqtrade/Nautilus
2. CCXT 是行业标准，生态成熟，文档完善
3. 自研方案可保持"回测=实盘"一致性
4. 与现有 `quant-sync-server` 架构兼容

## 技术架构

```
交易所 (Binance/OKX/Coinbase)
    ↓ REST API / WebSocket
数据采集层 (CCXT)
    ↓
历史数据库 (PostgreSQL)
    ↓
回测引擎 (自研)
    ↓
NEMT Core (信号生成)
    ↓
策略层 (信号→订单)
    ↓
执行层 (Paper/Live)
    ↓
云端部署 (Docker + 监控)
```

**技术栈**：
- 数据获取：CCXT 4.x
- 实时数据：CCXT Pro (WebSocket)
- 历史存储：PostgreSQL
- Web 服务：FastAPI 0.100+
- 容器化：Docker 24+
- 监控：Prometheus + Grafana
