## 开发规范

### Git 工作流
```
main ←←←←←←←←←←←←←←←
          ↖ feature/xxx
          |  (开发分支)
          |_______________
                    ↙ PR → CTO 审查 → 合并
```

### 分支命名
- `feature/xxx` - 新功能
- `fix/xxx` - Bug修复
- `refactor/xxx` - 重构

### Commit 规范
```
<类型>: <简短描述>

<可选的详细描述>

<可选的关联任务>
- Task #XXX
```

类型：
- `feat`: 新功能
- `fix`: Bug修复
- `docs`: 文档
- `refactor`: 重构
- `test`: 测试
- `chore`: 杂项

### 代码规范
- [ ] 遵循 PEP 8 (Python)
- [ ] 遵循 ESLint 规则 (JavaScript)
- [ ] 无硬编码配置
- [ ] 配置文件外置

## 单元测试规范

### 测试覆盖率目标
- 核心业务逻辑: ≥ 80%
- 工具函数: ≥ 90%
- API接口: ≥ 70%

### 测试命名
```python
def test_<function>_<scenario>_<expected>():
    """
    测试: <函数名>
    场景: <测试场景>
    预期: <期望结果>
    """
```

### 测试分类
- **单元测试**: 测试单个函数/方法
- **集成测试**: 测试多个组件协作
- **端到端测试**: 测试完整流程

## 代码审查响应

### 收到审查反馈时
- [ ] 仔细阅读每条反馈
- [ ] 区分阻塞性问题和建议性修改
- [ ] 修复所有问题
- [ ] 提交修复并通知 CTO

### 常见审查问题
| 问题类型 | 处理方式 |
|----------|----------|
| 逻辑错误 | 必须修复 |
| 安全漏洞 | 必须修复 |
| 性能问题 | 评估后决定 |
| 代码风格 | 建议修复 |
| 命名不清 | 建议修复 |

## 开发检查清单

### 功能完整性
- [ ] 实现了所有需求？
- [ ] 符合验收标准？
- [ ] 没有引入回归问题？

### 代码质量
- [ ] 代码清晰易读？
- [ ] 函数职责单一？
- [ ] 没有重复代码？

### 测试
- [ ] 单元测试覆盖核心逻辑？
- [ ] 所有测试通过？
- [ ] 新功能有对应测试？

### 配置
- [ ] 无硬编码？

- [ ] 环境变量处理正确？

- [ ] 需要更新的文档已更新？
- [ ] 复杂逻辑有注释？
