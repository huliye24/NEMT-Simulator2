## 一、核心原则（源自VISION.md）

### 1.1 策略是生物体原则

**原则**: 策略不是静态的规则集合，而是具有生命周期的实体

**技术解读**:
- 每个策略需要独立的配置文件
- 策略需要有ID、版本、状态标记
- 策略表现需要持续监控和评分
- 低分策略需要自动降权或淘汰

**代码要求**:
```python
# 每个策略必须继承策略基类
class BaseStrategy(ABC):
    @abstractmethod
    def generate_signal(self, market_data: MarketData) -> Signal:
        pass
    
    @property
    @abstractmethod
    def strategy_id(self) -> str:
        pass
    
    @property
    @abstractmethod
    def version(self) -> str:
        pass
```

### 1.2 适应优先于预测原则

**原则**: 系统目标是适应市场变化，而非预测市场走向

**技术解读**:
- 不使用黑盒预测模型（ML预测被禁止）
- 使用规则驱动的指标（DCI、涡旋、随机共振）
- 多策略组合分散风险
- 动态权重根据市场状态调整

**代码要求**:
- 禁止引入未经回测验证的预测模型
- 所有指标必须是可解释的
- 信号必须带有置信度

### 1.3 风控优先原则

**原则**: 任何情况下风险控制优先于追求收益

**技术解读**:
- 风控模块是不可绕过的
- 单笔交易风险必须硬限制
- 整体回撤超阈值必须自动降仓
- 紧急情况下一键清仓

**代码要求**:
```python
class RiskLayer:
    def check_order(self, order: Order) -> bool:
        """订单风控检查 - 不允许绕过"""
        if not self._check_single_risk(order):
            return False
        if not self._check_total_exposure(order):
            return False
        if not self._check_drawdown(order):
            return False
        return True
```

### 1.4 厨房理论

**原则**: 先有稳固架构，再添加功能

**技术解读**:
- 新功能必须先有接口和Mock实现
- 功能必须通过测试验证
- 代码必须符合规范才能合并
- 文档必须与代码同步更新

**开发流程**:
```
Kitchen阶段: 接口定义 → Mock实现 → 测试框架
Chef阶段:   接口实现 → 业务逻辑 → 集成测试
```

## 二、代码质量原则

### 2.1 可读性优先

**原则**: 代码是给人看的，顺便给机器执行

- 变量/函数命名必须清晰表达意图
- 复杂逻辑必须加注释
- 每个函数不超过50行
- 模块必须有docstring

### 2.2 模块化原则

**原则**: 高内聚、低耦合

- 每个模块有单一职责
- 模块之间通过接口通信
- 禁止循环依赖
- 使用依赖注入解耦

### 2.3 错误处理原则

**原则**: 优雅地处理错误

- 所有异常必须被捕获或显式声明
- 错误信息必须包含上下文
- 使用日志记录错误便于排查
- 关键操作需要重试机制

### 2.4 性能原则

**原则**: 除非必要，不做优化

- 先保证正确性，再考虑性能
- 性能问题必须有数据支撑
- 优化后必须回归测试
- 避免过早优化

## 三、测试原则

### 3.1 测试覆盖率要求

| 模块 | 覆盖率目标 |
|------|-----------|
| nemt_core.py | > 90% |
| nemt_signals.py | > 85% |
| nemt_state_machine.py | > 90% |
| nemt_risk.py | > 80% |
| nemt_execution.py | > 80% |
| 整体 | > 80% |

### 3.2 测试分类

```
单元测试 → 测试单个函数/类
集成测试 → 测试模块间协作
端到端测试 → 测试完整流程
```

### 3.3 Mock使用原则

- 外部依赖必须Mock
- 数据库操作必须Mock
- 网络请求必须Mock
- 时间相关操作可Mock

## 四、文档原则

### 4.1 必须文档化的内容

- 所有公共API必须有docstring
- 复杂算法必须添加解释
- 配置项必须有注释
- 架构变更必须更新ARCHITECTURE.md

### 4.2 文档更新时机

| 变更类型 | 文档要求 |
|----------|----------|
| 新增模块 | 必须同时创建文档 |
| 修改接口 | 必须更新接口文档 |
| 架构调整 | 必须更新架构图 |
| 新增依赖 | 必须更新README |

## 五、安全原则

### 5.1 密钥管理

- API密钥存储在环境变量
- 禁止硬编码密钥
- 敏感配置使用加密

### 5.2 运行时安全

- 策略执行超时保护（30秒）
- 仓位上限检查
- 单笔亏损自动熔断

### 5.3 数据安全

- 日志不记录敏感信息
- 用户输入必须验证
- SQL注入防护

## 六、DevOps原则

### 6.1 CI/CD

- 所有PR必须通过测试
- 代码必须通过lint检查
- 构建必须成功

### 6.2 部署

- 使用Docker容器化
- 配置与代码分离
- 支持滚动更新

### 6.3 监控

- 关键指标必须可观测
- 异常必须告警
- 日志必须结构化
