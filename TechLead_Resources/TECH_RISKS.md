## 五、风险评估标准

### 5.1 可能性评级

| 等级 | 描述 | 定义 |
|------|------|------|
| 高 | 很可能发生 | > 70% 概率 |
| 中 | 有可能发生 | 30% - 70% 概率 |
| 低 | 不太可能 | < 30% 概率 |

### 5.2 影响评级

| 等级 | 描述 | 定义 |
|------|------|------|
| 高 | 严重影响业务 | 导致核心功能不可用 |
| 中 | 中度影响 | 部分功能受限 |
| 低 | 轻微影响 | 可接受的用户体验下降 |

## 六、风险管理流程

```
1. 识别风险 → 2. 评估风险 → 3. 制定计划 → 4. 监控风险 → 5. 复盘
```

### 6.1 识别

- 日常代码审查中发现
- Sprint 回顾中发现
- 用户反馈中发现
- 技术扫描中发现

### 6.2 评估

使用风险矩阵:

```
              影响
         低    中    高
       ┌────┬────┬────┐
    高 │ P2 │ P1 │ P0 │
可能性  ├────┼────┼────┤
    中 │ P2 │ P2 │ P1 │
       ├────┼────┼────┤
    低 │ P2 │ P2 │ P2 │
       └────┴────┴────┘
```

### 6.3 监控

- 每月审查风险清单
- 高风险项每周跟踪
- 建立监控告警

## 七、风险缓解策略

### 7.1 规避

消除风险来源:
- 不使用不稳定的库
- 不依赖单点服务

### 7.2 转移

将风险转移:
- 购买保险
- 使用托管服务

### 7.3 缓解

降低风险影响:
- 增加测试覆盖
- 实现降级策略
- 准备回滚方案

### 7.4 接受

在可接受范围内:
- 记录风险
- 准备应急计划
- 持续监控
