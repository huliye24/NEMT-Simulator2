# 代码审查检查清单
> 每次 Code Review 必须逐项检查

## 一、CI 检查（必须通过）

- [ ] **测试通过**: `npm test` / `pytest` 无失败
- [ ] **Linter 通过**: ESLint / Pylint 无错误
- [ ] **类型检查**: TypeScript / mypy 无错误
- [ ] **构建成功**: 打包/编译无错误

## 二、代码质量

### 2.1 命名规范
- [ ] 变量/函数命名清晰表达意图
- [ ] 无拼音变量名（中文字符用于注释）
- [ ] 命名风格一致（camelCase/PascalCase/Snake_case）
- [ ] 无缩写歧义（如 `tmp` 应改为 `temporaryBuffer`）

### 2.2 函数设计
- [ ] 函数单一职责（SRP）< 50 行
- [ ] 无重复代码（DRY 原则）
- [ ] 参数数量 ≤ 4（超过需重构）
- [ ] 无副作用或副作用明确标注

### 2.3 错误处理
- [ ] 异常被捕获并有明确处理
- [ ] 关键操作有日志记录
- [ ] 错误消息对用户/开发者友好
- [ ] 无裸 `except:` 捕获

### 2.4 类型安全
- [ ] 有类型注解或 JSDoc
- [ ] 无 `any` 类型滥用
- [ ] 枚举用于有限集合
- [ ] 数据结构不可变性（`const`/`final`）

## 三、架构一致性

### 3.1 模块边界
- [ ] 无跨层直接调用（如 Service 直接调用 DAO 而非通过 Repository）
- [ ] 公共接口有文档
- [ ] 内部实现无泄露到公共 API

### 3.2 依赖管理
- [ ] 无循环依赖
- [ ] 依赖方向正确（高层依赖低层）
- [ ] 新增依赖必要且最小化

### 3.3 配置管理
- [ ] 硬编码值移到配置
- [ ] 环境变量用于敏感信息
- [ ] `.env.example` 更新（如有新增配置）

## 四、安全检查

- [ ] 无硬编码密钥/密码
- [ ] 用户输入已验证/消毒
- [ ] SQL/命令注入防护
- [ ] 权限检查到位

## 五、测试覆盖

### 5.1 单元测试
- [ ] 新增代码有测试
- [ ] 测试覆盖核心逻辑路径
- [ ] Mock 使用合理
- [ ] 测试命名清晰

### 5.2 测试质量
- [ ] 测试独立无依赖
- [ ] 测试可重复执行
- [ ] 无 `sleep` 或硬编码时间等待
- [ ] 断言有意义（非 `assertTrue(True)`）

## 六、性能考虑

- [ ] 无 N+1 查询问题
- [ ] 无循环内大内存分配
- [ ] 异步操作正确使用
- [ ] 大数据处理有分页/流式

## 七、审查决策

### 批准条件
所有检查项通过或有不影响功能的 minor 建议

### 需要修改条件
- [ ] 任何 blocker 级别问题
- [ ] 安全漏洞
- [ ] 测试缺失
- [ ] 架构违规

### 拒绝条件
- [ ] 故意绕过检查
- [ ] 恶意代码
- [ ] 剽窃代码
